En el presente capítulo se abordan los fundamentos teóricos necesarios para comprender el desarrollo de aplicaciones de cómputo paralelo mediante CUDA. \section{CUDA (Compute Unified Device Architecture)}

El crecimiento de aplicaciones científicas, de ingeniería y de análisis de datos ha incrementado la necesidad de utilizar arquitecturas capaces de procesar grandes volúmenes de información en tiempos reducidos. Aunque los procesadores tradicionales continúan siendo fundamentales para la ejecución de sistemas operativos y programas de propósito general, muchas tareas modernas presentan un alto grado de paralelismo que puede aprovecharse mediante hardware especializado. En este contexto surge CUDA (Compute Unified Device Architecture), una plataforma desarrollada por NVIDIA que permite utilizar la capacidad de procesamiento de las unidades gráficas para resolver problemas de cómputo general mediante programación paralela. Esta tecnología proporciona un conjunto de herramientas, bibliotecas y compiladores que facilitan el desarrollo de aplicaciones aceleradas por GPU utilizando extensiones de los lenguajes C y C++ \cite{nvidiaCUDA,nvidiaToolkit}.

\subsection{Definición y objetivo de la plataforma CUDA}

CUDA puede definirse como una arquitectura de cómputo paralelo y un modelo de programación diseñado para aprovechar los recursos de procesamiento disponibles en las GPU de NVIDIA. Antes de la aparición de esta tecnología, la utilización de tarjetas gráficas para cálculos científicos requería adaptar los problemas computacionales a interfaces gráficas como OpenGL o DirectX, lo que complicaba considerablemente el desarrollo de software. CUDA eliminó esta limitación al proporcionar una interfaz orientada al cálculo numérico, permitiendo que los desarrolladores trabajen con conceptos más cercanos a la programación tradicional y puedan acceder directamente a los recursos del dispositivo gráfico. Gracias a este enfoque, las GPU dejaron de utilizarse exclusivamente para tareas de renderizado y comenzaron a desempeñar un papel importante en áreas como simulación numérica, inteligencia artificial, minería de datos y procesamiento de imágenes \cite{nvidiaCUDA,investigacionCUDA}.

El propósito principal de CUDA consiste en proporcionar un entorno que permita distribuir una gran cantidad de operaciones entre miles de unidades de procesamiento disponibles en una GPU moderna. A diferencia de una CPU convencional, cuya arquitectura está optimizada para minimizar la latencia de ejecución de un número reducido de hilos, una GPU está diseñada para maximizar el rendimiento global mediante la ejecución simultánea de grandes cantidades de operaciones matemáticas. La plataforma CUDA abstrae gran parte de la complejidad asociada al hardware y ofrece mecanismos que facilitan la escalabilidad del software entre distintas generaciones de dispositivos, permitiendo que una misma aplicación pueda ejecutarse en tarjetas gráficas con diferentes capacidades de procesamiento sin modificaciones significativas en el código fuente \cite{nvidiaProgrammingGuide,investigacionCUDA}.

\subsection{Ventajas del uso de GPU para cómputo paralelo}

La utilización de GPU como aceleradores de cómputo ofrece diversas ventajas frente a los esquemas tradicionales basados únicamente en CPU. Una de las más importantes es la disponibilidad de miles de núcleos de procesamiento capaces de ejecutar instrucciones sobre diferentes conjuntos de datos de manera simultánea. Adicionalmente, las GPU cuentan con sistemas de memoria diseñados para proporcionar un ancho de banda considerablemente superior al disponible en la mayoría de las computadoras convencionales. Estas características permiten acelerar algoritmos que realizan operaciones repetitivas sobre grandes volúmenes de información, como multiplicación de matrices, simulaciones físicas, entrenamiento de redes neuronales o procesamiento de señales \cite{nvidiaCUDA,awsGPUCPU}.

\section{Arquitectura de una GPU}

La arquitectura interna de una GPU difiere considerablemente de la utilizada en los procesadores convencionales. Aunque ambos dispositivos son capaces de ejecutar instrucciones de propósito general, las decisiones de diseño implementadas en cada uno persiguen objetivos distintos. Las CPU modernas dedican una gran parte de sus recursos a mecanismos de control, predicción de saltos y memorias caché de gran tamaño con la finalidad de reducir la latencia de ejecución. Por el contrario, las GPU destinan la mayor parte de sus recursos físicos al procesamiento aritmético, permitiendo que una gran cantidad de operaciones sean ejecutadas de forma concurrente. Esta característica constituye la base del modelo de programación utilizado por CUDA \cite{awsGPUCPU,nvidiaProgrammingGuide}.

\subsection{Diferencias entre CPU y GPU}

Una diferencia fundamental entre ambos dispositivos se encuentra en la distribución de los transistores disponibles dentro del circuito integrado. En una CPU, una proporción considerable del hardware está dedicada a mecanismos que permiten optimizar la ejecución secuencial de instrucciones, incluyendo memorias caché multinivel, predictores de bifurcación y unidades de control avanzadas. Por el contrario, las GPU reducen la complejidad de estas estructuras y utilizan el espacio liberado para incorporar una mayor cantidad de unidades aritméticas. Como resultado, una GPU puede mantener activos miles de hilos de ejecución simultáneamente, ocultando los tiempos de espera asociados al acceso a memoria mediante cambios rápidos de contexto entre grupos de hilos.

Esta decisión permite obtener un mejor aprovechamiento de los recursos cuando se trabaja con problemas que pueden dividirse en una gran cantidad de operaciones independientes \cite{awsGPUCPU,investigacionCUDA}.

\subsection{Arquitectura interna de las GPU}

Los CUDA Cores constituyen las unidades básicas de procesamiento dentro de una GPU NVIDIA. Estos elementos son responsables de ejecutar operaciones aritméticas y lógicas sobre los datos asignados a cada hilo de ejecución. A diferencia de un núcleo completo de CPU, un CUDA Core no posee mecanismos avanzados de control ni capacidad para administrar procesos del sistema operativo. Su función principal consiste en realizar cálculos matemáticos de manera eficiente, permitiendo que miles de operaciones puedan ejecutarse simultáneamente sobre diferentes conjuntos de datos. En las arquitecturas modernas, estos núcleos trabajan junto con otros componentes especializados, como las unidades de funciones especiales y los Tensor Cores, para incrementar aún más el rendimiento en aplicaciones científicas y de inteligencia artificial.

Aunque los CUDA Cores son las unidades responsables de realizar las operaciones matemáticas, la organización principal del hardware se encuentra en los denominados Streaming Multiprocessors (SM). Cada SM integra un conjunto de recursos de procesamiento que incluye CUDA Cores, registros, memoria compartida, unidades de planificación y diversos mecanismos de control. Cuando un programa CUDA se ejecuta, los bloques de hilos son distribuidos entre los distintos SM disponibles en la GPU, permitiendo que múltiples tareas se desarrollen de forma paralela. Esta estructura jerárquica facilita la administración eficiente de los recursos y constituye uno de los elementos fundamentales para alcanzar altos niveles de rendimiento computacional \cite{introduccionCUDA,investigacionCUDA}.

El funcionamiento de las GPU modernas se basa en un modelo conocido como Single Instruction Multiple Threads (SIMT), mediante el cual un mismo conjunto de instrucciones puede ejecutarse simultáneamente sobre múltiples hilos. Dentro de cada Streaming Multiprocessor, los hilos se agrupan en conjuntos denominados warps, normalmente formados por treinta y dos hilos. Cada warp ejecuta una misma instrucción sobre datos diferentes, lo que permite aprovechar de manera eficiente las unidades aritméticas disponibles. Sin embargo, cuando algunos hilos siguen rutas de ejecución distintas debido a estructuras condicionales, el hardware debe serializar parcialmente la ejecución, reduciendo el rendimiento obtenido. Por esta razón, el diseño de algoritmos para CUDA suele buscar minimizar la divergencia entre los hilos pertenecientes a un mismo warp \cite{nvidiaProgrammingGuide,investigacionCUDA}.

\section{Modelo de programación CUDA}

La programación en CUDA se fundamenta en un modelo heterogéneo en el que participan dos tipos de procesadores con funciones complementarias. Por una parte, la CPU actúa como anfitrión del programa y se encarga de las tareas de control, administración de memoria y coordinación general de la aplicación. Por otra parte, la GPU funciona como dispositivo acelerador y ejecuta las secciones del código que presentan un alto grado de paralelismo. Este enfoque permite distribuir las responsabilidades de procesamiento entre ambos componentes, aprovechando las fortalezas de cada arquitectura y reduciendo significativamente el tiempo requerido para resolver problemas computacionalmente intensivos \cite{nvidiaProgrammingGuide,investigacionCUDA}.

A diferencia de los programas tradicionales escritos únicamente para CPU, las aplicaciones desarrolladas con CUDA requieren que el programador determine qué partes del algoritmo serán ejecutadas en el host y cuáles serán enviadas al dispositivo gráfico. Esta separación constituye uno de los aspectos más importantes del modelo de programación, ya que influye directamente en el rendimiento obtenido y en el aprovechamiento de los recursos disponibles en el hardware \cite{introduccionCUDA,nvidiaCUDA}.

\subsection{Modelo Host y Device}

El modelo Host-Device representa la base conceptual de CUDA. El término Host hace referencia al sistema principal donde se ejecuta la aplicación, normalmente una computadora equipada con una CPU convencional. El término Device corresponde a la GPU encargada de realizar las operaciones paralelas. Ambos componentes poseen espacios de memoria independientes, por lo que resulta necesario transferir información entre ellos cuando se requiere procesar datos en la tarjeta gráfica.

En una aplicación típica, la CPU reserva memoria, inicializa los datos de entrada y posteriormente realiza una transferencia hacia la memoria del dispositivo. Una vez completada la transferencia, la CPU solicita la ejecución de uno o más kernels en la GPU. Cuando los cálculos concluyen, los resultados son copiados nuevamente hacia la memoria principal para su utilización posterior. Este esquema permite aprovechar la capacidad de procesamiento masivo de la GPU sin reemplazar completamente las funciones de control desempeñadas por la CPU.

Mientras el host coordina la ejecución global de la aplicación, el device concentra sus recursos en realizar cálculos paralelos sobre grandes cantidades de datos. Esta distribución de responsabilidades constituye uno de los principios fundamentales del modelo de programación de CUDA \cite{hostdevice,nvidiaProgrammingGuide}.

\subsection{Concepto de Kernel}

Un kernel puede definirse como una función especial diseñada para ejecutarse directamente en la GPU. Cuando el host invoca un kernel, el hardware crea una gran cantidad de hilos que ejecutan simultáneamente el mismo conjunto de instrucciones sobre diferentes elementos de datos. Esta característica permite implementar algoritmos paralelos de forma relativamente sencilla, ya que el programador describe el trabajo que debe realizar un hilo individual y CUDA se encarga de replicar dicho comportamiento sobre miles de instancias concurrentes.

Los kernels se diferencian de las funciones tradicionales debido a que utilizan una sintaxis especial para su lanzamiento. Durante la ejecución, cada hilo dispone de información que le permite identificar su posición dentro de la estructura paralela, facilitando el acceso a diferentes posiciones de memoria y permitiendo distribuir el trabajo de forma organizada. Gracias a este mecanismo, problemas de gran tamaño pueden dividirse en numerosas tareas pequeñas que son ejecutadas simultáneamente por la GPU \cite{nvidiaProgrammingGuide,investigacionCUDA}.

\subsection{Flujo básico de ejecución de un programa CUDA}

El ciclo de ejecución de una aplicación CUDA sigue una secuencia claramente definida que inicia en la CPU y culmina con la recuperación de resultados procesados por la GPU. En primer lugar, el programa reserva memoria en el host y genera los datos necesarios para el cálculo. Posteriormente se asigna memoria en el dispositivo y se transfieren los datos requeridos hacia la GPU. Una vez completada esta etapa, el host lanza el kernel correspondiente y espera la finalización de la tarea. Finalmente, los resultados obtenidos son copiados nuevamente hacia la memoria principal para continuar con el procesamiento o presentarlos al usuario.

Este procedimiento resulta indispensable debido a que la memoria de la GPU y la memoria principal del sistema se encuentran físicamente separadas. Como puede observarse en la Figura \ref{fig:flujo_cuda}, la ejecución de una aplicación CUDA implica la transferencia de información entre el host y el dispositivo antes y después del procesamiento paralelo realizado por la GPU. Como consecuencia, la eficiencia de una aplicación CUDA no depende únicamente de la velocidad de cálculo del dispositivo, sino también de la cantidad de transferencias realizadas entre ambos espacios de memoria. Una adecuada planificación de estas operaciones permite reducir tiempos de ejecución y mejorar el aprovechamiento del hardware disponible \cite{nvidiaProgrammingGuide,introduccionCUDA}.

\section{Organización de hilos en CUDA}

Uno de los elementos más importantes de CUDA es el mecanismo utilizado para organizar los miles de hilos que participan en una ejecución paralela. Debido a que una GPU moderna puede ejecutar simultáneamente una enorme cantidad de tareas, resulta necesario establecer una estructura jerárquica que permita distribuir el trabajo de manera ordenada. CUDA implementa esta organización mediante tres niveles principales denominados Thread, Block y Grid, los cuales facilitan la administración de recursos y simplifican el desarrollo de algoritmos paralelos escalables \cite{threadHierarchy,investigacionCUDA}.

Antes de analizar el mecanismo de indexación de hilos, resulta conveniente estudiar la jerarquía de ejecución utilizada por CUDA para organizar el trabajo dentro de la GPU. Este modelo establece diferentes niveles que permiten distribuir las tareas de manera eficiente entre los recursos de procesamiento disponibles. Cada nivel cumple una función específica dentro de la ejecución paralela, desde la realización de operaciones individuales hasta la administración de grandes conjuntos de trabajo. Gracias a esta organización jerárquica, CUDA puede escalar la ejecución de un mismo kernel sobre grandes cantidades de datos manteniendo una estructura ordenada y flexible. Las características principales de cada nivel se presentan en la Tabla \ref{tab:jerarquia_cuda}.

Como resultado, una misma aplicación puede aprovechar desde unos pocos hilos hasta millones de ellos, dependiendo de las dimensiones del problema y de las capacidades del hardware disponible \cite{threadHierarchy,nvidiaProgrammingGuide}.

\subsection{Identificación de hilos}

Para que cada hilo pueda determinar qué parte de los datos debe procesar, CUDA proporciona variables integradas que identifican su posición dentro de la jerarquía de ejecución. Entre las más utilizadas se encuentran \texttt{threadIdx}, \texttt{blockIdx}, \texttt{blockDim} y \texttt{gridDim}, las cuales son generadas automáticamente por el sistema de ejecución. Estas variables permiten conocer la ubicación de cada hilo dentro de su bloque, así como la posición del bloque dentro de la grilla. Gracias a esta información, es posible asignar de manera ordenada las operaciones que realizará cada hilo y establecer una correspondencia directa entre los elementos de los datos y las unidades de procesamiento disponibles. Este mecanismo constituye uno de los aspectos fundamentales del modelo de programación de CUDA, ya que permite distribuir el trabajo de forma eficiente entre miles de hilos concurrentes \cite{threadIndexing,investigacionCUDA}.

En problemas unidimensionales, el índice global de un hilo suele obtenerse mediante la Ecuación \ref{eq:indice_global}, la cual combina la posición del bloque dentro de la grilla con la posición del hilo dentro del bloque para generar un identificador único. Este índice permite asociar cada hilo con una posición específica dentro de un arreglo lineal almacenado en memoria, facilitando el acceso a los datos que deben ser procesados. A partir de este esquema, cada hilo puede ejecutar las mismas instrucciones sobre diferentes elementos sin interferir con los demás, lo que favorece la ejecución paralela de grandes volúmenes de información. Debido a ello, el cálculo del índice global representa uno de los conceptos más utilizados en el desarrollo de algoritmos paralelos mediante CUDA \cite{threadIndexing,threadCheatsheet}.

\begin{equation}
\label{eq:indice_global}
idx = blockIdx.x \times blockDim.x + threadIdx.x
\end{equation}

Mientras que algunas regiones ofrecen una gran capacidad de almacenamiento a costa de una mayor latencia, otras proporcionan accesos extremadamente rápidos pero con recursos más limitados. El aprovechamiento adecuado de esta estructura permite minimizar los retrasos asociados a la transferencia de información y mantener ocupadas las unidades de procesamiento de la GPU durante el mayor tiempo posible. En conjunto, estos espacios de memoria permiten equilibrar capacidad, velocidad y alcance de acceso, contribuyendo al desarrollo de aplicaciones paralelas con un mejor aprovechamiento de los recursos de la GPU \cite{cudaMemoryHierarchy,nvidiaBestPractices,cornellMemory,nvidiaProgrammingGuide}.

\subsection{Registros}

Cada hilo dispone de un conjunto privado de registros utilizados para almacenar variables temporales y resultados intermedios durante la ejecución. Debido a que estos recursos se encuentran integrados directamente dentro del multiprocesador, las operaciones de lectura y escritura pueden realizarse prácticamente en un único ciclo de reloj. No obstante, la cantidad de registros disponibles es limitada y debe compartirse entre todos los hilos activos en un multiprocesador. Un uso excesivo de registros puede reducir la cantidad de bloques concurrentes que pueden ejecutarse simultáneamente, afectando la ocupación del dispositivo y disminuyendo el rendimiento global de la aplicación. \subsection{Importancia de la jerarquía de memoria en el rendimiento}

La correcta utilización de la jerarquía de memoria constituye uno de los factores más importantes para alcanzar altos niveles de rendimiento en CUDA. Un acceso desordenado a la memoria global puede generar múltiples transacciones innecesarias y aumentar significativamente los tiempos de ejecución. Por el contrario, cuando los datos son organizados adecuadamente y reutilizados mediante memoria compartida o registros, es posible reducir la latencia efectiva y mejorar el aprovechamiento de los recursos computacionales.

Entre las estrategias más utilizadas para optimizar el acceso a memoria destacan la coalescencia de accesos en memoria global y la reducción de conflictos de bancos en memoria compartida. Estas técnicas permiten incrementar el ancho de banda efectivo disponible para los kernels y constituyen prácticas ampliamente recomendadas por NVIDIA para el desarrollo de aplicaciones de alto rendimiento \cite{nvidiaBestPractices,cudaMemoryHierarchy}.

\section{Compilación y ejecución de programas CUDA}

El desarrollo de aplicaciones CUDA requiere un proceso de compilación diferente al utilizado en programas tradicionales escritos únicamente para CPU. Debido a que una aplicación CUDA contiene código destinado a ejecutarse tanto en el host como en el dispositivo, es necesario utilizar herramientas capaces de generar instrucciones para ambas arquitecturas de manera simultánea. Para este propósito, NVIDIA proporciona el compilador NVCC (NVIDIA CUDA Compiler), el cual actúa como intermediario entre el código fuente desarrollado por el programador y los compiladores específicos encargados de generar los ejecutables finales. La principal función de NVCC consiste en analizar los archivos fuente, identificar las secciones destinadas a la CPU y las destinadas a la GPU, y posteriormente dirigir cada una hacia el compilador correspondiente. Gracias a este mecanismo, el desarrollador puede trabajar dentro de un mismo proyecto utilizando código secuencial y paralelo sin necesidad de emplear herramientas de compilación independientes. Este enfoque simplifica el desarrollo de aplicaciones heterogéneas y permite aprovechar de forma eficiente los recursos de hardware disponibles en el sistema \cite{nvccGuide,nvidiaProgrammingGuide,investigacionCUDA}.

\subsection{Compilador NVCC y proceso de compilación}

NVCC es el compilador oficial de CUDA y constituye uno de los componentes fundamentales del entorno de desarrollo proporcionado por NVIDIA. Su función principal consiste en coordinar el proceso de traducción del código fuente hacia instrucciones ejecutables tanto para la CPU como para la GPU. Para ello, NVCC analiza los calificadores específicos de CUDA, identifica los kernels y genera los archivos intermedios necesarios para completar el proceso de compilación. A diferencia de un compilador convencional, NVCC no realiza por sí solo toda la generación del código máquina. En su lugar, utiliza compiladores externos compatibles con la plataforma para procesar la parte correspondiente al host, mientras que emplea herramientas especializadas de NVIDIA para generar las instrucciones destinadas al dispositivo gráfico. Esta estrategia permite mantener compatibilidad con distintos sistemas operativos y arquitecturas de hardware.

El proceso de compilación de una aplicación CUDA puede dividirse en varias etapas. Inicialmente, NVCC analiza el código fuente e identifica las regiones correspondientes al host y al dispositivo. Posteriormente, el código destinado a la GPU es traducido a PTX (Parallel Thread Execution), un lenguaje ensamblador virtual diseñado por NVIDIA para representar instrucciones de manera independiente del hardware específico. Una vez generado el código PTX, herramientas internas del compilador producen los archivos binarios optimizados para una arquitectura determinada de GPU. Finalmente, estos componentes se integran junto con el código compilado para CPU dentro de un único ejecutable. Este procedimiento permite que las aplicaciones CUDA aprovechen las capacidades específicas del hardware disponible sin perder portabilidad entre diferentes generaciones de dispositivos \cite{nvccGuide,nvidiaProgrammingGuide}.

\subsection{Archivos utilizados y estrategias de compilación}

Estas extensiones facilitan la organización del código y proporcionan información al compilador acerca de la naturaleza de cada archivo. Los archivos con extensión \texttt{.cu} contienen la implementación principal de los programas CUDA y pueden incluir tanto código para CPU como kernels destinados a la GPU. Por otra parte, los archivos \texttt{.cuh} se utilizan generalmente para almacenar declaraciones, prototipos y estructuras compartidas entre diferentes módulos del proyecto. Durante el proceso de compilación también se generan archivos intermedios que representan distintas etapas de traducción y optimización del código. Asimismo, CUDA admite distintos mecanismos para generar código ejecutable. Uno de ellos es la compilación offline, donde las instrucciones específicas para una arquitectura determinada son generadas durante el proceso de desarrollo. En este caso, el ejecutable contiene código optimizado para una familia concreta de dispositivos gráficos, reduciendo el tiempo requerido para iniciar la aplicación. Este enfoque proporciona una mayor compatibilidad con dispositivos futuros, ya que permite generar código optimizado para arquitecturas que no existían al momento de desarrollar el programa original \cite{nvccGuide,nvidiaProgrammingGuide}.

\section{Propiedades de dispositivos CUDA}

Esta información resulta útil para adaptar parámetros como el tamaño de los bloques de hilos, la cantidad de memoria utilizada y la selección de determinadas estrategias de procesamiento. De esta manera, las aplicaciones pueden aprovechar de forma más eficiente los recursos disponibles y garantizar la compatibilidad con diferentes generaciones de hardware.
